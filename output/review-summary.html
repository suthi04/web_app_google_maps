1. บทสรุปผู้บริหาร (Executive Summary)

ระบบที่พัฒนาจริง ทำงานได้และมีคุณภาพวิศวกรรมสูงกว่าที่เล่มบรรยายไว้มาก — มี pipeline ครบ (scrape → คัดไทย → clean → tokenize → sentiment → aspect → phrase extraction → insights → dashboard), มีชุดทดสอบ 28 ไฟล์/~135 เทสต์, มีโมดูลประเมินผลเชิงปริมาณ และ WangchanBERTa ใช้งานได้จริง (รายงานผล 88.3% accuracy)

ปัญหาหลักไม่ใช่โค้ดบกพร่อง แต่คือ “เล่มวิจัยกับโค้ดเล่าคนละเรื่องกันบางส่วน”:

  1. [Critical] เล่มอ้างว่า fine-tune WangchanBERTa ด้วยข้อมูลรีวิวร้านอาหาร (หน้า 23) — โค้ดไม่มีการ fine-tune ใด ๆ ใช้ checkpoint สำเร็จรูป finetuned@wisesight_sentiment แบบ off-the-shelf
  2. [Critical] หัวใจของระบบจริงคือ การสกัด opinion phrase เชิงพจนานุกรม + Gemini LLM + การจัดหมวด 4 ชั้น + insights — ทั้งหมดนี้ไม่ปรากฏในเล่ม
  3. [Critical] เล่มไม่เอ่ยถึง Gemini/LLM แม้แต่คำเดียว ทั้งที่เป็น 1 ใน 2 extraction engine
  4. [Important] ค่าเริ่มต้นของระบบคือ lexicon ไม่ใช่ WangchanBERTa (USE_MODEL=0, extract_engine="rule") ขัดกับการที่เล่มชู WangchanBERTa เป็นพระเอก
  5. [Critical] บทที่ 4 (ผลการวิจัย) และบทที่ 5 ยังเป็น template เปล่า ทั้งที่ eval/report.txt มีผลพร้อมใช้แล้ว
  6. [Minor] เล่มระบุจัดเก็บเป็น CSV — โค้ดใช้ SQLite (insightreview.db) + export CSV
สรุป: งานวิศวกรรมแข็งแรง แต่เอกสารวิจัย “ต่ำกว่าความจริง” ในส่วนที่เป็นนวัตกรรมจริง และ “เกินความจริง” ในเรื่อง fine-tuning — ทั้งสองทิศทางคือความเสี่ยงในห้องสอบ

7. ความเสี่ยงเชิงวิชาการ (Academic Risks)

  1. อ้าง fine-tune แต่ไม่มีในโค้ด → ขอดู training script แล้วไม่มี = เสียความน่าเชื่อถือทั้งเล่ม
  2. บทที่ 4–5 ว่าง = ไม่ผ่านเกณฑ์เนื้อหา
  3. Gemini/LLM ไม่อยู่ในเล่ม → กรรมการเปิดโค้ดเจอ = หาว่าปิดบัง
  4. default lexicon vs ชู WangchanBERTa → สาธิตสดได้ผลคนละชุด
  5. contribution หลัก (phrase) ไม่มี quantitative evaluation
  6. ชุดประเมิน balance เทียม → ตัวเลขสวยเกินจริง
  7. ไม่มี error analysis / neutral อ่อน
  8. CSV↔SQLite, JavaScript ตกหล่นจากขอบเขต
